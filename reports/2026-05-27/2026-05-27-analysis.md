## 1. Executive Summary

- **v4 (generated column + B-tree index) is the clear winner for known-path searches.** MySQL's internal execution time on a covering index lookup is **~0.02ms**. Whatever latency the application sees on top of that is network / connection overhead, not query cost.
- **v5 (FULLTEXT) is viable for free-text body search.** Internal execution stays under 35ms even when matching 4,669 rows. Worth pursuing if the product needs "search the whole body for any keyword".
- **v1 (CAST + LIKE) and v2 (JSON_SEARCH) are not competitive.** Both do full table scans with per-row JSON parsing. P95 = 1s on a 45k-row tenant with no date filter, and will scale linearly with row count.
- **v3 (`->>` path extract without an index) is also a full table scan** and gives no real benefit over v1/v2 — its only value is as the "before" baseline for v4's "after".
- **The data has known caveats — see §2 before drawing conclusions from absolute latency numbers.**
- **Write benchmark is not informative in its current form** (see §2.2). A re-run with batched INSERTs is needed before write-side trade-offs (especially the v5 FULLTEXT write penalty) can be quantified.

## 2. Data Caveats (read this before §3)

Three measurement artifacts limit how literally the raw numbers should be read.

### 2.1 Read benchmark is bottle-necked by network RTT (~310 ms)

The DB is remote. EXPLAIN ANALYZE proves it directly:

```
v5 missing keyword, MySQL-internal execution time:
  -> Full-text index search ... (actual time=0.006..0.006 rows=0 loops=1)

But the application-observed P50: 317.6 ms.
```

Every measured query carries ≈ 310 ms of network + driver overhead before the actual SQL work even matters. **Absolute P50/P95/P99 values in this run reflect "wire time", not "query time"**. The scheme-to-scheme comparison is still valid because every scheme pays the same network tax, but the *magnitude* of differences is compressed.

To recover the approximate "true MySQL time", subtract 310 ms from the absolute number. We provide both columns in §3 so you can see both.

### 2.2 Write benchmark is unusable in absolute terms

```
v1: 3.1 inserts/sec
v2: 3.1 inserts/sec   (-0.6% vs v1)
v3: 3.1 inserts/sec   (-1.3%)
v4: 3.1 inserts/sec   (-1.1%)
v5: 3.0 inserts/sec   (-2.2%)
```

All five schemes report essentially the same throughput. **This is the network RTT again** — 2000 serialized `INSERT … VALUES (…)` round-trips × 310 ms ≈ 620 s, which matches the observed 644-658 s. The actual write cost of each scheme is buried under the network overhead.

Action: the benchmark script needs to switch to **batched INSERTs** (e.g. 50 rows per `INSERT`) before write-side conclusions can be drawn. The v5 FULLTEXT write penalty in particular is the open question.

### 2.3 Storage column reports `0 B`

`information_schema.tables.data_length` is cached for 24 hours by default in MySQL 8 (`information_schema_stats_expiry = 86400`). The query was run shortly after table creation, so it returned the initial value. Not a real "tables are empty" signal.

Action: the script needs to either set `information_schema_stats_expiry = 0` for the session or call `ANALYZE TABLE` before reading the size.

### 3.2 What `EXPLAIN ANALYZE` says about each scheme

The strongest signal in this run is the MySQL-internal times reported by `EXPLAIN ANALYZE`, because network is not in the picture.

**v1 — full table scan with per-row JSON-to-CHAR coercion:**

```
-> Filter: (cast(...requestBody as char ...) like '%string%')
   (cost=7448 rows=32697) (actual time=0.794..531 rows=2153 loops=1)
    -> Index lookup using audit_logs_tenant_index (tenant='lotusflaretelecom')
       (actual time=0.032..124 rows=45024 loops=1)
```

Tenant index narrows it down to 45k rows, then every row is parsed and pattern-matched. 531 ms is the cost on 45k rows; on a million-row tenant it would be ~12 s.

**v2 — full table scan with JSON_SEARCH:**

```
-> Filter: (json_search(... '%string%') is not null)
   (cost=6543 rows=28711) (actual time=0.507..355 rows=2115 loops=1)
    -> Index lookup using audit_logs_tenant_index (rows=45024)
```

355 ms inner cost. Slightly cheaper than v1 because it avoids the full CAST, but same fundamental shape — linear in tenant row count.

**v3 — same scan + JSON path extract:**

```
-> Filter: (json_unquote(json_extract(...,'$.id')) = '...')
   (cost=7168 rows=31505) (actual time=132..132 rows=1 loops=1)
    -> Index lookup using audit_logs_tenant_index (rows=45024)
```

132 ms inner cost. Substantially cheaper than v1/v2 because the comparison is a single short string per row, but **still full scan** — the optimizer cannot index the path expression.

**v4 — covering index on the generated column (this is the winner):**

```
-> Covering index lookup on audit_logs_bench_v4
   using idx_reqBody_id_tenant
   (tenant='lotusflaretelecom', reqBody_id='cb4a86a6-…')
   (cost=1.04 rows=1) (actual time=0.018..0.021 rows=1 loops=1)
```

0.021 ms — three orders of magnitude faster than v3 on the same logical query, because the generated column is indexed.

**v5 — FULLTEXT inverted-list walk:**

```
common keyword "string":
-> Filter: (... tenant='...' and match against ('string' in boolean mode))
   (actual time=3.61..32.9 rows=2080 loops=1)
    -> Full-text index search ... (actual time=2.28..32.1 rows=4669 loops=1)

missing UUID:
-> Full-text index search ... (actual time=0.006..0.006 rows=0 loops=1)
```

Common-keyword cost is 32 ms (long inverted list with 4669 candidates); missing keyword is 0.006 ms (inverted index lookup, no hits, exits immediately). This is **healthy FULLTEXT behavior**.

### 3.3 Effect of `dateTime` pre-filter

For scan-based schemes, narrowing the date window dramatically reduces work:

| Scheme | `all` P95 | `7d` P95 | `24h` P95 | Speedup (all → 24h) |
|---|---:|---:|---:|---|
| v1 common | 1.06 s | 360 ms | 350 ms | ~3× |
| v2 common | 948 ms | 357 ms | 343 ms | ~3× |
| v3 `$.id` | 482 ms | 337 ms | 352 ms | ~1.4× |
| v4 `$.id` | 358 ms | 364 ms | 340 ms | ~1× (already index-fast) |

After subtracting the ~310 ms network floor: v1's true work drops from ~750 ms to ~40 ms when constrained to 24h, because there are only ~130 rows in that window. **A "must pick a date window first" UX could rescue v1/v2 for short-range searches, but not for full-history searches.**

### 3.4 v1 vs v2 — v2 is not meaningfully better

Across all scenarios, v2 (`JSON_SEARCH`) is at best ~15% faster than v1 (`CAST + LIKE`), often slower. The two schemes have essentially the same execution shape (full scan + per-row JSON inspection). **v2 has no product value and can be dropped from the candidate set.**

### 3.5 P99 tail observations on v4

v4's `$.companyId` query shows P95 = 506 ms, P99 = 1.01 s, despite EXPLAIN proving the inner cost is 0.03 ms. This is **network jitter** — a handful of measurements got unlucky. In production with low-latency MySQL connectivity, v4 should consistently return in <5 ms.

## 4. Write Benchmark Analysis

### 4.1 Why the numbers can't be trusted in their current form

See §2.2. The benchmark issues 2000 serialized single-row INSERTs over a remote connection, so 95% of the wall time is round-trip latency. All five schemes end up looking identical at 3.1 inserts/sec.

### 4.2 What we still can confirm

The EXPLAIN evidence and basic understanding of MySQL guarantee a few things even without good throughput numbers:

- **v4 incurs one extra B-tree maintenance per indexed JSON path on every INSERT.** With 3 paths (`$.id`, `$.orderId`, `$.companyId`), that's 3 extra B-tree updates per write. In practice that's a single-digit percent overhead.
- **v5 incurs FULLTEXT inverted-index maintenance on every INSERT.** This is tokenization + inverted-list update for every word in the JSON. Historically this is the most expensive write penalty in the MySQL toolkit — often -30% to -50% throughput in literature.
- **v1, v2, v3 add no write overhead** (no schema change).

**Quantifying v4 and v5's true write penalty requires re-running with batched INSERTs.** This is a known follow-up.

### 4.3 Storage cost — also a follow-up

Same root cause as §2.3. We expect v5 to roughly double the on-disk size (original JSON column + stringified column + FULLTEXT index), and v4 to add a small overhead per indexed path (~5-10% per path). The re-run will confirm.

## 5. Cross-cutting Findings

### Confirmed
- ✅ **v4 generated-column + B-tree is fully effective** as long as the JSON path is known in advance.
- ✅ **v5 FULLTEXT works correctly** on this data shape and is the only viable scheme for free-text body search.
- ✅ **v1 / v2 / v3 cannot be made fast enough** for full-history search at production-scale row counts.
- ✅ **A date-window pre-filter is a strong product knob** for scan-based schemes (3× speedup for v1/v2 at 24h).
- ✅ **v2 (JSON_SEARCH) offers no meaningful improvement over v1** and should be removed from the recommendation set.

### Open (requires re-run with fixed scripts)
- ❓ **Real write-throughput penalty of v4 (estimated single-digit %) and v5 (estimated 30-50%).**
- ❓ **Actual storage overhead of v4 generated columns and v5 FULLTEXT index.**
- ❓ **Behavior on a "fat" tenant with body sizes closer to the 2.57 MB max** (this run's sample tenant has avg ~1.2 KB).

## 6. Recommendation

### 6.1 Recommended scheme combination

| User intent | Recommended scheme | Justification |
|---|---|---|
| Look up by specific `requestId` | (existing `requestId` B-tree index) | already covered, no scheme change needed |
| "Find logs where `body.subscriberId = X`" or any other known field | **v4** with that path indexed | MySQL-internal cost ~0.02 ms; reuses standard B-tree |
| "Find logs containing string `X` anywhere in body" | **v5** with FULLTEXT | only sub-second free-text option in MySQL |
| `body.field LIKE '%substring%'` (substring of a sub-token) | Not feasible in MySQL at scale | only v1 truly satisfies the semantics, but v1 doesn't meet SLA |

### 6.2 Product UX implication

Free-text body search (v5) does **token-based** matching, not character-substring matching. This is a UX difference from how column search currently works and **must be flagged to the PM**. Specifically:

- `MATCH('abc')` finds documents containing the token `abc`; it does **not** find documents containing the substring `bc` inside a longer token.
- Tokens shorter than 3 characters are silently ignored (MySQL's default `innodb_ft_min_token_size`).
- Hyphens / underscores / punctuation split tokens.

Most user search cases are well-served by token matching, but it should be explicit in the UI.

### 6.3 SLA target — to confirm with PM

This benchmark was run against a remote dev MySQL with ~310 ms RTT. Production latency will be substantially lower. As a placeholder for PM confirmation:

- For known-path search (v4): **P95 ≤ 50 ms** is realistic.
- For free-text search (v5): **P95 ≤ 500 ms** is realistic for the current data size; will need re-validation as audit_logs grows past ~10 M rows.
