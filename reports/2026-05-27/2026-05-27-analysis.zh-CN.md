# Audit Log Body 搜索 benchmark — 分析报告

**测试日期**：2026-05-27
**作者**：基于 `run-2026-05-27-09-11-26`（读）和 `run-2026-05-27-10-23-24`（写）整合
**测试环境**：MySQL 8.4.8，数据库 `portal`，表 `audit_logs`（读跑了 76,575 行 / 写跑了 76,635 行）
**采样 tenant**：`lotusflaretelecom`（约 45,050 行，是行数最多的 tenant）
**v3 / v4 测试用的 JSON path**：`$.id`、`$.orderId`、`$.companyId`

> 🇬🇧 English version: [2026-05-27-analysis.md](./2026-05-27-analysis.md)

---

## 1. 执行摘要

- **已知 path 的精确搜索：v4（生成列 + B-tree 索引）完胜。** MySQL 内部覆盖索引查找耗时 **~0.02ms**。应用层看到的延迟全部是网络 / 连接开销，跟 query 本身无关。
- **Free-text body 搜索：v5（FULLTEXT）可行。** 即使命中 4669 行，FULLTEXT 内部耗时 ≤35ms。如果产品需要"搜整个 body 任意关键词"，这是唯一选择。
- **v1（CAST + LIKE）和 v2（JSON_SEARCH）不行。** 两者都是全表扫 + 每行 JSON 解析。45k 行 tenant + 不限时间范围，P95 = 1 秒；随行数线性增长。
- **v3（`->>` path 提取但无索引）也是全表扫**，对 v1/v2 没有实质改进——它的价值在于给 v4 做对比基线（"加索引前"vs"加索引后"）。
- **数据有 3 个已知局限性 —— 看完 §2 再下结论**。
- **写 benchmark 当前版本不可用**（见 §2.2）。需要修改成 batch INSERT 后重跑，才能量化写入侧的代价（特别是 v5 FULLTEXT 的写惩罚）。

## 2. 数据局限性（看 §3 之前必须先看）

三个测量瑕疵限制了"绝对数字"的解读方式。

### 2.1 读 benchmark 被网络 RTT（~310ms）卡住

数据库是远程的。EXPLAIN ANALYZE 直接证明：

```
v5 missing 关键词，MySQL 内部执行时间：
  -> Full-text index search ... (actual time=0.006..0.006 rows=0 loops=1)

但应用层观察到的 P50：317.6 ms。
```

每个测量到的 query 在真正干活之前都先付出了 ~310 ms 的网络 + 驱动开销。**这次跑出来的绝对 P50/P95/P99 反映的是"传输时间"，不是"query 时间"**。方案之间的对比仍然有效（因为每个方案都付同样的网络税），但**差异的绝对幅度被压缩了**。

要恢复大致的"MySQL 真实耗时"，把绝对数字减去 310 ms。§3 里两栏数据都给了，方便对照。

### 2.2 写 benchmark 数据完全不可用

```
v1: 3.1 inserts/sec
v2: 3.1 inserts/sec   (-0.6% vs v1)
v3: 3.1 inserts/sec   (-1.3%)
v4: 3.1 inserts/sec   (-1.1%)
v5: 3.0 inserts/sec   (-2.2%)
```

5 个方案几乎一样的吞吐。**还是网络 RTT 的问题**——2000 次串行 `INSERT … VALUES (…)` × 310 ms ≈ 620 秒，跟实测的 644-658 秒吻合。每个方案真实的写入代价被网络开销完全掩盖。

待办：脚本需要改成 **batch INSERT**（比如每次 INSERT 50 行）后，写入侧结论才有意义。特别是 v5 FULLTEXT 的写惩罚是必须回答的开放问题。

### 2.3 Storage 显示 `0 B`

MySQL 8 的 `information_schema.tables.data_length` 默认有 24 小时缓存
（`information_schema_stats_expiry = 86400`）。我们在建表后不久就查，
返回的是初始值，**不是"表是空的"**。

待办：脚本里要么 `SET SESSION information_schema_stats_expiry = 0`，
要么在读 size 前先 `ANALYZE TABLE`。

## 3. 读 benchmark 分析

### 3.1 延迟概览（采样 tenant，不限时间范围）

下表把观察到的 P95 和 `EXPLAIN ANALYZE` 报告的 `actual time` 放在一起对比——后者是 MySQL 内部成本，没有网络开销。

| 方案 | 关键词 / path | 命中行 | 观察 P95 | EXPLAIN 内部成本 | 推测的 query 实际耗时（P95 − 310ms） | 结论 |
|---|---|---:|---:|---:|---:|---|
| v1 | common（`string`）| 2,153 | 1.06s | 531ms | ~750ms | ❌ 随表大小线性增长 |
| v1 | rare（`Service_with_…`）| 0 | 1.01s | 634ms | ~700ms | ❌ 不管命中多少都一样慢 |
| v1 | missing（UUID）| 0 | 988ms | 536ms | ~680ms | ❌ |
| v2 | common | 2,115 | 948ms | 355ms | ~640ms | ❌ 比 v1 略好但仍然全表扫 |
| v2 | missing | 0 | 770ms | 377ms | ~460ms | ❌ |
| v3 | `$.id` = UUID | 1 | 482ms | 132ms | ~170ms | ⚠️ 无索引但 path 提取本身不贵 |
| v3 | `$.companyId` = '136' | 7 | 482ms | 135ms | ~170ms | ⚠️ |
| **v4** | `$.id` = UUID | 1 | 358ms | **0.02ms** | ~50ms（受网络限制）| ✅ **索引完美工作** |
| **v4** | `$.companyId` = '136' | 7 | 506ms | 0.03ms | ~200ms（P99 网络抖动，见 §3.5）| ✅ 索引工作，注意抖动 |
| **v5** | common（`string`）| 2,080 | 489ms | 32ms | ~180ms | ✅ FULLTEXT 32ms 走完 4669 文档 ID |
| **v5** | rare | 0 | 362ms | 0.007ms | ~50ms | ✅ 几乎瞬间 |
| **v5** | missing | 0 | 337ms | 0.006ms | ~25ms | ✅ 几乎瞬间 |

### 3.2 EXPLAIN ANALYZE 告诉我们的方案差异

这次跑里最可信的信号是 `EXPLAIN ANALYZE` 报告的 MySQL 内部时间，因为没有网络干扰。

**v1 — 全表扫 + 每行 JSON 转 CHAR：**

```
-> Filter: (cast(...requestBody as char ...) like '%string%')
   (cost=7448 rows=32697) (actual time=0.794..531 rows=2153 loops=1)
    -> Index lookup using audit_logs_tenant_index (tenant='lotusflaretelecom')
       (actual time=0.032..124 rows=45024 loops=1)
```

tenant 索引把范围缩到 45k 行，然后每一行都被解析并做 pattern 匹配。45k 行花了 531ms；如果一个 tenant 涨到 100 万行，会变成约 12 秒。

**v2 — 全表扫 + JSON_SEARCH：**

```
-> Filter: (json_search(... '%string%') is not null)
   (cost=6543 rows=28711) (actual time=0.507..355 rows=2115 loops=1)
    -> Index lookup using audit_logs_tenant_index (rows=45024)
```

内部 355ms。比 v1 略便宜（少了完整的 CAST），但**形态完全一样**——线性于 tenant 行数。

**v3 — 同样全表扫 + JSON path 提取：**

```
-> Filter: (json_unquote(json_extract(...,'$.id')) = '...')
   (cost=7168 rows=31505) (actual time=132..132 rows=1 loops=1)
    -> Index lookup using audit_logs_tenant_index (rows=45024)
```

内部 132ms。比 v1/v2 显著便宜（每行只比较一个短字符串），但**仍然是全表扫**——优化器不会在 path 表达式上建索引。

**v4 — 生成列上的覆盖索引（最优解）：**

```
-> Covering index lookup on audit_logs_bench_v4
   using idx_reqBody_id_tenant
   (tenant='lotusflaretelecom', reqBody_id='cb4a86a6-…')
   (cost=1.04 rows=1) (actual time=0.018..0.021 rows=1 loops=1)
```

0.021ms，比 v3 在相同逻辑查询上**快了三个数量级**，因为生成列有索引。

**v5 — FULLTEXT 倒排链表扫描：**

```
common 关键词 "string"：
-> Filter: (... tenant='...' and match against ('string' in boolean mode))
   (actual time=3.61..32.9 rows=2080 loops=1)
    -> Full-text index search ... (actual time=2.28..32.1 rows=4669 loops=1)

missing UUID：
-> Full-text index search ... (actual time=0.006..0.006 rows=0 loops=1)
```

common 关键词 32ms（4669 个候选的长倒排链表）；missing 关键词 0.006ms（倒排索引查无后立即退出）。**这是 FULLTEXT 健康的表现**。

### 3.3 dateTime 预过滤的效果

对扫描型方案，缩小时间窗口能显著降低工作量：

| 方案 | `all` P95 | `7d` P95 | `24h` P95 | 加速（all → 24h） |
|---|---:|---:|---:|---|
| v1 common | 1.06s | 360ms | 350ms | ~3× |
| v2 common | 948ms | 357ms | 343ms | ~3× |
| v3 `$.id` | 482ms | 337ms | 352ms | ~1.4× |
| v4 `$.id` | 358ms | 364ms | 340ms | ~1×（已经索引快了）|

减去 ~310ms 网络后：v1 的真实耗时从 ~750ms 降到 ~40ms（24h 范围内只有 ~130 行）。**如果 UX 强制"必须先选时间窗口才能搜 body"，v1/v2 在短时间范围下勉强可用，但全历史搜不行**。

### 3.4 v1 vs v2 — v2 没有实质改进

所有场景下，v2（`JSON_SEARCH`）最多比 v1（`CAST + LIKE`）快 ~15%，常常更慢。两者本质上是同一种执行形态（全表扫 + 每行 JSON 检查）。**v2 没有产品价值，可以从候选方案中删除。**

### 3.5 v4 的 P99 尾延迟现象

v4 在 `$.companyId` 查询上 P95 = 506ms、P99 = 1.01s，但 EXPLAIN 证明内部成本只有 0.03ms。这是**网络抖动**——几次测量运气不好。生产环境低延迟连接下，v4 应该稳定 <5ms 返回。

## 4. 写 benchmark 分析

### 4.1 为什么当前数据不可信

见 §2.2。脚本通过远程连接发 2000 次串行单行 INSERT，95% 的墙钟时间都是 RTT 占用。所以 5 个方案最后看上去都是 3.1 inserts/sec。

### 4.2 即使没有可信数据，仍能确认的几件事

EXPLAIN 证据 + 对 MySQL 的基本理解，已经能保证以下几个事实：

- **v4 每个 INSERT 多付一次 B-tree 维护代价 × 已建索引的 JSON path 数。** 3 个 path（`$.id`、`$.orderId`、`$.companyId`）就是每次写多 3 次 B-tree 更新。实际上是个位数百分比的开销。
- **v5 每个 INSERT 会触发 FULLTEXT 倒排索引维护。** 即每个 JSON 词都要做分词 + 倒排链表更新。这是 MySQL 工具箱里写惩罚最大的索引——文献里常见 -30% 到 -50% 的吞吐下降。
- **v1、v2、v3 没有任何写入额外开销**（没有 schema 改动）。

**量化 v4 和 v5 真实的写惩罚需要 batch INSERT 重跑**。这是确定的待办。

### 4.3 存储成本 — 同样需要重跑

根因同 §2.3。预期 v5 会让磁盘大小翻倍（原 JSON 列 + 字符串化列 + FULLTEXT 索引），v4 每个 path 增加约 5-10%。重跑后会确认。

## 5. 综合结论

### 已确认
- ✅ **v4 生成列 + B-tree 完全有效**，前提是 path 事先约定好。
- ✅ **v5 FULLTEXT 在这份数据上工作正确**，是 free-text body 搜索的唯一可行方案。
- ✅ **v1 / v2 / v3 在生产规模下无法做到全历史搜索可接受的速度。**
- ✅ **时间窗口预过滤对扫描型方案是强力的产品手段**（v1/v2 在 24h 范围下提速 3×）。
- ✅ **v2（JSON_SEARCH）对 v1 没有实质改进**，应该从推荐方案中剔除。

### 待确认（需要修复脚本后重跑）
- ❓ **v4 的真实写惩罚（估计个位数 %）和 v5 的真实写惩罚（估计 30-50%）。**
- ❓ **v4 生成列和 v5 FULLTEXT 索引的实际存储开销。**
- ❓ **在 body 大小接近 2.57 MB 上限的"胖" tenant 上的表现**（本次采样 tenant 平均 ~1.2 KB）。

## 6. 推荐结论

### 6.1 推荐方案组合

| 用户场景 | 推荐方案 | 理由 |
|---|---|---|
| 用某个 `requestId` 查日志 | 已有 `requestId` B-tree 索引 | 已经够用，不需要新方案 |
| "找 `body.subscriberId = X` 这类已知字段的查询" | **v4**，给这个 path 加索引 | MySQL 内部成本 ~0.02ms；用的还是标准 B-tree |
| "找 body 里任意位置包含字符串 X 的日志" | **v5** FULLTEXT | MySQL 内唯一亚秒级 free-text 选项 |
| `body.field LIKE '%substring%'`（子 token 的子串）| MySQL 规模下不可行 | 只有 v1 真满足这个语义，但 v1 不满足 SLA |

### 6.2 产品 UX 影响

Free-text body 搜索（v5）是**token-based 匹配**，不是字符 substring 匹配。这跟现有 column search 的体验不同，**必须告知 PM**。具体：

- `MATCH('abc')` 找含 `abc` 这个 token 的文档；**不会**找到含 `bc` 作为长 token 的子串的文档。
- 短于 3 个字符的 token 会被静默忽略（MySQL 默认 `innodb_ft_min_token_size`）。
- 连字符 / 下划线 / 标点会切分 token。

大多数用户场景 token 匹配都够用，但 UI 上要明确说明。

### 6.3 SLA 目标 — 待 PM 确认

本次 benchmark 是在远程 dev MySQL 上跑的（RTT ~310ms）。生产环境延迟会显著更低。作为 PM 确认前的占位值：

- 已知 path 搜索（v4）：**P95 ≤ 50ms** 现实可行。
- Free-text 搜索（v5）：**P95 ≤ 500ms** 在当前数据规模可行；audit_logs 涨到 ~1000 万行后需要重新验证。
